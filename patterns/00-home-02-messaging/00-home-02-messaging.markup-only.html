Messaging Guidelines

Supporting Principles
  • Colour should not be the only element used to convey information
  • Using standard bootstrap classes for alerts will ensure that text and background have a high enough contrast to comply with the W3C standards for color contrast

Feedback messages can be given in three ways: Help Blocks, Inline Alerts, or Modal Dialogs.

Use Help Blocks:
  • To associate messages with a particular element or section of the screen.
  • To display feedback messages (i.e. error, warning, success) related to validating user input.
  • To give tips for inputting information in a form element
Use Inline Alerts:
  • To inform the user of a situation without interrupting their focus.
  • To associate a message with a specific section of a page.
  • To give unintrusive feedback about an action just performed
Use Modal Dialogs:
  • To interrupt the users' workflow to inform them of a situation or prompt them for input.
  • To confirm changes in flow or direction, for example to confirm the movement from one page or form to another.

  • Should clearly communicate the message as succictly as possible
  • Can specify a small text size if vertical space is a concern
Help Blocks
  • Should be placed inside a .form-group, placed immediately following input fields with a class of .form-control
  • Should not include icons as we do not want any added visual emphasis for help text
  • Can be made contextual by adding the .text-info, .text-success, .text-warning, or .text-danger classes. For example:
    • This is an informational message
    • This is an success message
    • This is an warning message
    • This is an error message
Validation Messages
  • Will automatically be formatted using the appropriate colors when the .form-group has the class .has-feedback and one of .has-error, .has-warning, or .has-success applied.
  • Use a glyphicon (glyphicon-ok, glyphicon-warning-sign, or glyphicon-remove) with a class of .form-control-feedback to display an icon on the right side of the input field

Usage Guidelines for Inline Alerts:

All inline alerts should:

  • Use proper grammar and punctuation.
  • Avoid the use of slang and/or technical jargon.
  • Use language that is business-like and friendly

Success Alerts Should:
  • Appear immediately following an action performed by the user.
  • Show a brief and succinct message to inform the user that an action completed successfully.
  • Should automatically dismiss themselves after a period of time (typically 5 seconds).
Information Alerts Should:
  • Present information that the user may want to be aware of.
  • Be informative only and not require any direct action on the part of the user
  • Should remain visible until the message is dismissed.
Warning Alerts Should:
  • Not be too strongly worded or imply blame.
  • State the message, briefly and succinctly.
  • Should remain visible until the message is dismissed.
Error Alerts Should:
  • State that an error was encountered, briefly and succinctly.
  • Be free of blame - the message should indicate that the system has encountered a problem, not the user.
  • Should remain visible until the message is dismissed.
Usage Guidelines for Modal Dialogs:

Use a Modal Dialog when you want to:

  • Display important information, as a way to prevent or correct critical errors.
  • Request the user to enter information critical to continuing the current process.
  • Fragment a complex workflow into simpler steps.
Wording Guidelines:
  • The modal header should display 1-3 short attention getting words in the header that hint at the purpose of the modal.
  • The modal body should show a short succinct description of the situation.
  • The modal footer should contain buttons for responding to the information presented.
  • Default Modal Dialogs:
    • Ask a question in the header (such as "Are you sure?").
    • Word the question so that Yes, the default button, is the natural response.
    • If the button that executes the modal states an action word (such as delete, add, move, or assign), then the confirmation should use the same action word. For example...
      • Header Text: Are you Sure?
      • Confirmation Text: Are you sure you want to delete the selected items?
      • Button Text: Yes, Delete selected
    • Position the buttons so that the negative response is on the left and the positive response on the right
    • The no or negative response should be the less likely choice, and positioned on the left.
    • The yes or positive response should be the most likely choice, and be positioned on the right.
