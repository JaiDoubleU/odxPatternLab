Communication Protocols

All integration communication with the OpenInvoice Platform is enabled with HTTPS and two-way certificate-based authentication. The transmitted messages are XML-based. XML files carry data in a self-descriptive structure. The XML files may have a SOAP, MIME or RNIF wrapper for the purposes of routing and security. MIME and RNIF wrappers add the ability to include attachments in the file transfer.

Acknowledgments

The OpenInvoice Platform will post directly to the client back-end system using RNIF with two-way SSL conversation. The conversation will be asynchronous with the return of an HTTP Status Code as 202. This means the message was successfully received, understood, and accepted. OpenInvoice will then wait for the Acknowledgment or Exception from the receiver. A Positive Acknowledgment indicates the business document has been successfully processed. A Negative Acknowledgment or Exception indicates the receiver fails to process the document.

Whenever a file in any format, except RNIF, is imported into the OpenInvoice Platform, OpenInvoice sends out an acknowledgment file in return. There are two kinds:

·Synchronous: When a file enters the OpenInvoice Platform, OpenInvoice acknowledges that the file has been received within the same conversation (connection between computers). HTTP status codes indicate the success or failure of the document transfer. A successful transaction will be indicated with the return of an HTTP Status Code in the 200 class. This means the message was successfully received, understood, and accepted. A failed transaction will be indicated with the return of an HTTP Status Code in the 400 or 500 classes.

·Asynchronous: For certain types of file transfers, a new conversation will be initiated and OpenInvoice will post a message back to your company indicating whether OpenInvoice has processed (imported) the request successfully.

Errors and Warnings

An error is a condition that prevents successful import or export of a document. If an error is present in the response, the document cannot be imported or exported. The error must be resolved and the document must be resent. For example, errors in documents may be a result of the XML not validating against its designated schema or the platform cannot establish the identity of the supplier or buyer.

Warnings present possible issues, but they will not prevent a document from importing or exporting. If the possible issues are not corrected, the same warnings will appear on subsequent imports and exports. For example, warnings may appear if multiple price book names are referenced on an invoice or if a referenced AFE on an invoice is not listed as one of the buyer's AFEs.

Retries on Imports

Imports may fail due to:

·Broken connectivity between the OpenInvoice Platform and the client.

·The business document fails on schema or business rules.

If a document update batch import fails, OpenInvoice will send a DOResponse.xsd schema within the SOAP body of the DeliveryInformation referencing the original post. A Failure status in the main part of the DOResponse indicates that the file was not processed. No information is known about the content. Once the problem is rectified, the client may resubmit the file.

Retries on Exports

Exports may fail due to:

·Broken connectivity between the OpenInvoice Platform and the client server.

·The exported file cannot be saved on the secure server.

·The client back-end URL is down.

The number of OpenInvoice export attempts will depend on the type of file. Most exports are configured to allow five attempts with a five minute interval between each attempt. Ticket exports may have a different frequency.

If all attempts are unsuccessful, the file is parked. OpenInvoice Support receives a parked status report from Trading Networks (integration software) at regular intervals so they can work with the appropriate contact to investigate and resolve communication issues.

Troubleshooting

OpenInvoice Support can view all transmissions in Trading Networks. Integration messages (such as purchase order XML files, field ticket XML files, etc.) will persist in Trading Networks for 45 days in production and seven days in the test environment for retrieval by OpenInvoice Support for troubleshooting. Attachments are not kept in Trading Networks.
