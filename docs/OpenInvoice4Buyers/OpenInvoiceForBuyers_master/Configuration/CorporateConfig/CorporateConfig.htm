Organizational Structure and Overview

All companies transacting in the OpenInvoice Platform are structured into sites, departments, and roles. This structure helps control user access to different functional areas of the platform; much like how employees perform office tasks based on their roles and responsibilities, OpenInvoice users are restricted to only those areas in the platform that pertain to their job functions.

Your company consists of one or more sites. Inside each of these sites are one or more departments. Inside each department are one or more roles, and each role has a different set of permissions.

In the example below, the company has two sites. Each site has two departments. Each department has two roles.

 

·Sites: geographical locations where a company operates

§Subsidiaries: similar to a site but is a separate legal entity within a company; the subsidiary name is displayed on the invoice instead of the company name. This option displays on the Add Site page.

·Departments: subdivisions of a site defined by the various functional areas of a company

·Roles: user groups defined by the specific work tasks they perform within a department

·Permissions: characteristics of roles and departments that allow users to perform their work tasks

·Licenses: features assigned to provide user access to particular modules in OpenInvoice

·Users: employees assigned to any number of roles across one or more departments and sites to perform their work tasks

Initial Company Setup

OpenInvoice Support initially structures and configures your company in the OpenInvoice Platform with:

·your company name (future company name changes will be updated in the platform by OpenInvoice Support)

·at least one administrative user with Membership and Corporate permissions who will internally complete your company’s setup

·at least one site and department

·purchased product licenses

·bank account information if using OpenInvoice to make payments

·integration configuration which includes an optional secure server and electronic document job configuration for automatic importing and exporting of files

·custom labels on the user interface which display the terminology used by your users in their everyday business practices.

Unregistered suppliers cannot access the OpenInvoice Platform. Their company details are entered into OpenInvoice by one of your users who has the Corporate Partnership permission.

Configuring Sites

This section provides descriptions of the fields associated with site configuration and describes how to add, edit, deactivate, and delete sites.

Site Profile Details

This section provides the descriptions of the fields on the site profile page.

·Site Name: the name used to identify the site in OpenInvoice and on all documents.

·Site Code: a code defined by your company’s ERP system. A site code is required when working with image invoice suppliers.

·Organization Type: will be Site or Subsidiary. Selecting Subsidiary replaces the company name on invoices with the site name.

·Location: further describes the site name.

·Active: indicates if the site is active.

·Comments: additional comments regarding the site.

·Primary Address and Primary Contact: address information that is displayed on document headers. Contact information is for internal purposes and, therefore, not displayed.

How to Add a Site

Use this procedure to add a new site profile.

Permission:Corporate Admin

Steps:

1.Select Corporate > Sites.

2.Click Add Site.

 

3.Enter the site’s profile information.

 

4.Click Submit.

5.Select Membership > Departments & Roles from the main menu.

6.Click Add Department.

 

7.Enter Department Name, Site, and Notification user. The other fields are optional.

Note:To search for a notification user, enter all or part of the user’s name and click the Search icon.

8.Click Submit.

 

9.Click Member under organization roles.

10.Select one or more Non-Member users, then click Add >.

11.Select the checkbox under each permission you want to assign to the Member role.

12.Click Add Role.

 

13.Enter Role Name and Description (optional).

14.Select Non-Members, then click Add > to add the users as members.

15.Click Submit.

16.Select the checkbox under each permission you want to assign to the new role.

Configuring Company Preferences

Note:Some company preferences are Oildex-controlled and require additional documentation and approval before they can be configured. User access to invoice data fields may be configured to be hidden, view-only, or editable. Make sure users have the ability to edit all required fields.

General Preferences

·Enable AFE or Cost Center Ownership: AFEs and Cost Centers can each be assigned owner. Owners will be displayed on the forwarding screen of invoices pending approval and one of them will be the default owner. AFE and Cost Center approval levels may apply when approving an invoice.

Note:Cost object owners need to have the Invoice Approve permission to be included in the forwardee list of an invoice pending approval. This feature is not available when forwarding multiple invoices at once.

·Enable Supplier Categorization: allows users with the Corporate Partnership permission to assign a supplier type for each enabled supplier.

·Enable Supplier Merge (Oildex-controlled): allows merging of unregistered supplier sites and registered supplier sites with the same supplier number. All invoices and purchase orders from the unregistered supplier site are transferred. Once merging is complete, the unregistered supplier site is deleted.

·Enable Units of Measure Compliance: makes the units of measure field on an invoice line required for all direct entry and B2B suppliers. The actual units are entered under Reference Data. Direct entry suppliers must select a unit of measure value from the buyer’s predefined list. Invoices submitted from B2B suppliers will be disputed automatically on the buyer’s behalf if one of the invoice lines does not include a valid unit of measure value. Invoices submitted before the preference is enabled are not affected. The preference has no effect on receipts.

·Company Assistance URL: defines the address of a web page that your users are directed to if they have trouble accessing OpenInvoice. This URL will be included in messages sent to user email addresses during attempts to recover access.

·Number of Custom Authorization Types (Oildex-controlled): defines the maximum number of approval types which can be mapped to general ledger major codes.

·Skin and Label Customization Identifier (Oildex-controlled): customized OpenInvoice labels that correspond with the terminology used by buyer companies in their everyday business practices. This customized terminology is visible to suppliers where appropriate.

Security Preferences

·Days to Password Expiry: defines the number of days that a user can use the same password before being prompted to change it. Allowed values: an integer in the range 30 to 360. Default Value: 180.

·Duration of Account Lockout: defines the number of minutes that locked out users must wait before attempting to sign in again. If the preference is set to indefinite, an email notification is sent to all users with the Membership Admin permission, and a user with the Membership Modify permission must manually unlock the locked out user. Allowed values: indefinite, 5, 10, 15 or 30. Default value: 5.

·Minimum Number of Hours Users Allowed Between Password Changes: defines the number of hours users must wait after changing their own password before they can change it again. Allowed values: an integer in the range 24 to 999,999. Default value: 24.

·Number of Previous Passwords Disallowed for Re-Use: defines the number of historical passwords (including the current password) the system uses when validating a new password for a user. A new password cannot be in the historical list. Allowed values: an integer in the range of 4 to 999,999. Default value: 4.

·Number of Successive Failed Sign In Attempts before Account Lockout: defines the number of times a user can enter an incorrect password before being locked out of their account. The user will receive a message to contact support on their next login attempt. Allowed values: an integer in the range 3 to 8. Default value: 3.

·Keep Me Signed In Enabled: makes the Keep Me Signed In checkbox visible on the Sign In page. This feature is available to all users within your company.

·Keep Me Signed In - Maximum Time Between Uses: sets the maximum time frame between uses before a user’s token expires and they must manually sign in again. Allowed values: 1 hour, 4 hours, 8 hours, 1 day, 2 days, 1 week, 2 weeks, 30 days. Default value: 2 days.

·Keep Me Signed In - IP Address Restriction Type: specifies when users must be connected from a specified IP address in order to use the Keep Me Signed In feature. Allowed values: Never, On Opt-In, Always. Default value: Never.

·Keep Me Signed In - Allowed IP Addresses: manages the allowable IP addresses from which a user can use the Keep Me Signed In feature.

Invoice Content Preferences

·Enable Additional Line Item Description: allows the buyer to enter an invoice line description in addition to the description entered by the supplier. A gray “DA” alert is displayed on lines with a buyer description. Moving the mouse over the alert will display the buyer’s description.

·Require AFE or Cost Center: makes the AFE and Cost Center fields required on an invoice line. A supplier must enter a valid value for one of the two fields before the line can be added to the invoice. This preference has no effect on B2B suppliers.

·Require PO: makes the purchase order number field on a detail invoice line required. Both purchase orders in the OpenInvoice Platform and external purchase orders are allowed. Invoices submitted from B2B suppliers will be disputed automatically on the buyer’s behalf if one of the invoice lines does not include a purchase order number. Image invoices are not affected.

·Require Invoice Approver: adds the Invoice Approver field to the invoice header and makes it required for direct entry suppliers. Displays the approver’s name and phone number as a contact in the supplementary information section of the invoice view screen for both suppliers and buyers. This preference has no effect on B2B suppliers. Buyer users who have been assigned an authorization level and an invoice permission in at least one department will populate the list of search results for the suppliers.

·Require Invoice Line Service Dates: makes the service date field(s) required on a detail invoice line and makes the service date field required on an image invoice. The Service Date field on a detail invoice header remains optional. Invoices submitted from B2B suppliers will be disputed automatically on the buyer’s behalf if any of the invoice lines do not include both service dates.

·Restrict Field/Lease by AFE or Cost Center: restricts the Field/Lease on an invoice coding record to the Field/Lease associated with the specified AFE or cost center.

·Restrict Location by AFE or Cost Center: restricts the Location on an invoice coding record to the Location associated with the specified AFE or cost center.

Invoice Workflow Preferences

·Allow Forwarding to Invoice View-Only Users: allows buyers to forward invoices to users with view-only permissions.

·Days Following Approved Export Before Disabling Amend: defines the maximum number of days following the approved invoice export that a user with the Invoice Amend permission is able to modify the general ledger coding of an approved invoice. By default this is not active, so there is no time limit on when an Amend action can occur. If this feature is enabled the allowed value is an integer in the range 0 to 999999.

·Days to Cancel Disputed Invoices: defines the number of days that must pass before a disputed invoice is automatically canceled. Allowed values: an integer in the range 90 to 9999. Image Post and Image OBO invoices submitted by buyers cannot be automatically canceled.

·Days to Stalled Invoice After Last Action: defines the number of days following the last action before an invoice is considered stalled. Adds Stalled counters to the invoice dashboard. Stalled invoices are indicated in the Invoice List page by yellow highlighting in the Type column. Allowed values: an integer in the range 0 to 999999.

·Days to Stalled Invoice After Most Recent Submit: defines the number of days following an invoice submission (or resubmission) from the supplier before the invoice is considered to be stalled. Stalled counters on the invoice dashboard are not affected by this preference. Stalled invoices are indicated on the Invoice page by red highlighting in the Type column.

·Disable Amend After Post Action: prevents users with the Invoice Amend permission from modifying the general ledger coding of an approved invoice after the invoice has been updated as Posted through an integrated response.

·Disable Mandatory Comments on Invoice Forward Action: allows users to forward an invoice without having to enter a comment.

Note:Comments are always optional when forwarding an invoice as part of the Code Verify or Approval Workflow actions.

·Enable Alert for Duplicate Ticket No. Reference: enables a red ticket alert when a field ticket number is referenced on more than one invoice. The alert appears on the invoice line and next to the invoice number on the Invoice List pages.

·Enable Dispute Categorization: This feature is currently inactive and the user’s selection is not referenced again in the application. The Comments field is still mandatory when disputing an invoice.

·Enable Stalled Documents Email Notification: sends an email notification of a stalled invoice to both the invoice owner and the department’s notification user.

·Enable Supplier Correction: allows users with the Invoice Modify (IM) permission to modify the supplier and/or supplier site for an image invoice using the Correct feature.

·Invoice Approval - Requisitioner May Not Final Approve Invoice: makes the Requisitioner field available on a detail invoice line and the Image OBO invoice indexing page. If a requisitioner is specified then an invoice manually approved by the requisitioner must be forwarded to another user for additional approval.

·Invoice Approval - Absolute Value Gross Line Amount: makes invoice approval and invoice workflow routing based on the absolute value of all invoice lines (credit lines are added to debits instead of subtracted).

·Invoice Counts Date Type: sets the date type for the Invoice Counts dashboard section. The selected date type is used to calculate the invoice counts categorization.

·Invoice Due Date Period: sets the number of days after submission that an invoice is due.

·Invoice Due Date Type: sets the date type used to calculate the invoice due date. Options are: Invoice Date or Submitted Date.

·Parse Format for Supplier Coding: defines the string mask (parse format) used to translate the specified general ledger code into the corresponding major, minor, and subcode values for invoices and tickets submitted from B2B suppliers. The supplier provided coding must match the defined format in order to be considered valid. If the preference is not enabled then the supplier provided coding will be considered valid if it matches any configured accounting string, regardless of format.

·Populate Smart-Coding From Invoice: selects the stage in the workflow where an invoice’s line item coding will be used to automatically update the smart coding. When Approved: smart coding rules updated upon invoice approval. When Code Verified: smart coding rules updated upon code verification. Never: never automatically update smart coding. The never setting requires updates to be made manually or through an import/export process. Default value: When Approved.

·Disable OBO Invoice Processing Emails (Oildex-controlled): turns off the feature that automatically sends an email notification to an unregistered supplier whenever an OBO ivoice is submitted on their behalf.

·Enable OBO Invoice Tax Fields: enables the Total Tax field and the Tax Type drop-down menu on the Image Invoice Processing page.

·Enable Invoice Number Masking (Oildex-controlled): generates a second invoice number that contains only alphanumeric characters (A-Z, 0-9) with a maximum length of 16 characters. Invoices will have both the original invoice number provided by the supplier and the masked number generated by OpenInvoice. Only the masked invoice number is included in the invoice exports. Both  invoice numbers are used for duplicate invoice number checking, and either invoice number can be used when searching for an invoice.

·Hide Unowned Invoices Dashboard Bin (Oildex-controlled): hides the Unowned Invoices dashboard category in the Invoices portlet.

·Ignore PO Mismatch on B2B Invoice Import (Oildex-controlled): allows B2B suppliers to submit invoices if the supplier site listed is not one of the supplier sites on the referenced purchase order.

·Invoice Approval – Company Preferred Approval Currency (Oildex-controlled): establishes a single currency to which authorization levels are defined. The invoice currency is ignored when determining if a user has the sufficient authorization level.

Note:If this is not configured, then the approver must have an authorization level in the currency of the invoice.

·Invoice Approval Method (Oildex-controlled): determines the type of authorization level required by a financial approver.

§GL Based Approvals: the financial approver must have an authorization level for every general ledger code on the invoice. This is the default setting.

§Total Invoice Amount: any general ledger codes on the invoice are ignored for approval purposes and the financial approver only needs a Total authorization level.

·Invoice Workflow Sequence (Oildex-controlled): determines the sequence of events when processing invoices in the OpenInvoice Platform.

§Code Then Approve: every invoice must be code verified before it is financially approved. This is the default setting.

§Approve Only: invoices can be financially approved without a separate code verification.

§Approve then Code: every invoice must be financially approved before it is code verified.

·Prevent Supplier Attachments after Submit (Oildex-controlled): If this preference is selected, suppliers will not be able to add attachments to submitted invoices (that have not been disputed by the buyer).

Ticket Workflow Preferences

·Approve Tickets When Approving an Invoice: allows a buyer to auto-approve all unapproved Field Tickets associated with an invoice when approving an invoice.

·Enable Multiple Superintendents, Office Approvers on the Same Invoice: allows a supplier to create an invoice from multiple tickets having different superintendents and office approvers. If this is not active, the system will not allow you to combine tickets with different superintendents or office approvers on the same invoice.

·Enforce Invoice Coding Validation on Field Ticket Approval: allows buyers to enforce the invoice coding validation rules when approving a field ticket.

Purchase Order Preferences

·PO Coding Override: when enabled this preference allows users to override the coding provided on a purchase order.

·Allow Blank PO Reference(s): when enabled this preference allows users to create and/or code an invoice with some lines referencing a purchase order and some lines with no purchase order reference. Any “Purchase Order Required” rules are satisfied if at least one line on the invoice has a purchase order reference.

·Allow Multiple PO Reference(s): when enabled this preference allows users to reference multiple purchase orders on a single invoice.

·Enable PO Counter Offers: allows suppliers to create counter offers against a purchase order.

·Prevent Invoice Submission on RED PO Compliance Alerts: prevents a direct entry supplier from submitting a detail invoice with a red purchase order compliance alert. This preference has no effect on image invoices or B2B suppliers.

·Prevent Receipt Submission on RED PO Compliance Alerts: This preference is obsolete and is no longer supported. It is scheduled to be removed from the platform in a future release.

·Require Explicit PO Line: disables the purchase order line number option No PO Line on invoices. All lines on a detail or itemized image invoice referencing a purchase order must also reference a valid line on that purchase order. Invoices submitted from B2B suppliers will be disputed automatically on the buyer’s behalf if one of the invoice lines does not include a purchase order line reference.

·Invoice Price Book Must Match Purchase Order: controls how strict the match must be between the price book on the invoice and the price book on the purchase order. When the preference is active, the invoice must refer to the same price book as the purchase order, including no price book when the purchase order does not refer to one. When the preference is not active, the system will still default the invoice price book to match that of the purchase order, but the supplier can refer to any valid price book.

Price Book Preferences

·Days to Display Inactive/Expired Price Books: defines the number of days that expired, cancelled, and terminated price books are included in the Price Book Dashboard. Allowed values: an integer in the range 1 to 999999.

·Days to Notify in Advance of Price Book Expiry: defines when a price book administrator will be notified in advance of a price book’s expiration. Allowed values: an integer in the range 1 to 365. Default value: 90.

Note:In addition to the notification sent on the configured number of days before the price book expires, OpenInvoice will automatically send a notification one day before the price book expires.

·Amber Price Book Alert Overrides Blue (Oildex-controlled): Price Book alerts are categorized by severity: red, yellow (amber), or blue. By default, red is the most severe, blue is the second most severe, and yellow is the least severe. When enabled, this preference modifies alert severity so that yellow takes priority over blue; red is most severe, yellow is second most severe, and blue is least severe.

·Enable Co-Managed Catalogs (Oildex-controlled): Allows buyers to create catalog items on a Direct-Entry supplier's behalf through a TSV import.

Active vs Inactive Custom Reference Data

Custom reference data fields can be set to ‘Active’ or ‘Inactive’ through the user interface, TSV import, and the API for Custom Reference Data.

By default all existing Custom Reference Data records are set to ‘Active.’ If you do not want a new Custom Reference Data record to be active you must uncheck the Active option when the record is created.

‘Inactive’ Custom Reference Data does not appear in the drop-down list when adding invoice header information, line item details, or line item coding information. ‘Inactive’ Custom Reference Data cannot be entered or searched for when using the Searchable Text display type.

Reference Data Types

·Custom Reference Data: data fields that are customized to correspond to your daily business terminology and practices. These fields can be added to invoice headers, line item details, or line item coding. Custom reference data fields can be set to ‘Active’ or ‘Inactive’ through the user interface, TSV import, and the API for Custom Reference Data. Records are set to ‘Active’ by default unless explicitly set to inactive. Inactive records will not appear in drop-down lists and cannot be entered or searched for.

·AFE Type: categorization of AFEs into types, which will save time on configuration and maintenance (coding) if you have a large number of AFEs.

·Approval Type: custom approval requirements created by mapping to general ledger codes which are then added to authorization levels. A general ledger code can only belong to one approval type at a time.

·Coding Level 1 – Major: main general ledger codes used to classify types of expenditures. The Cost Type is hard-coded to Capital or Expense. Code and description combinations must be unique for each general ledger coding level. The general ledger codes added here must be mapped to establish valid major.minor.subcode combinations. If a general ledger code or description is changed after the codes have been mapped, the mappings do not change.

·Coding Level 2 – Minor: minor general ledger codes used to sub-classify expenditures within majors accounts. Code and description combinations must be unique for each general ledger coding level. The general ledger codes added here must be mapped to establish valid major.minor.subcode combinations. If a general ledger code or description is changed after the codes have been mapped, the mappings do not change.

·Coding Level 3 – Subcode: subcodes used to sub-classify expenditures within minor accounts. Subcodes are optional unless a user sets them as mandatory for specific general ledger codes. Code and description combinations must be unique for each general ledger coding level. The general ledger codes added here must be mapped to establish valid major.minor.subcode combinations. If a general ledger code or description is changed after the codes have been mapped, the mappings do not change.

·Cost Center Type: categorization of cost centers into types, which will save time on configuration and maintenance (coding) if you have a large number of cost centers. Cost centers and cost center types can be configured so that only specified general ledger codes are available when coding an invoice associated with a specific cost center/cost center type. When configuring the restricted general codes for a cost center type, all cost centers assigned to that type will automatically inherit the coding restrictions.

·Dispute Type: categorization of reasons for invoice disputes.

·Field/Lease: classification of properties and assets. These can be associated with one or more AFEs and cost centers so the Field/Lease code gets added to the general ledger string that is exported to your ERP.   

·Purchase Category: purchase categories that are mapped to major general ledger codes. The list will appear in the drop-down list on invoices.

·Supplier Type: categorization of supplier sites to set up automatic invoice approvals. Types can be based on the workflow and business documents used to transact with the supplier (such as PO, Non-PO, Image) or the types of services performed by the supplier (such as Courier, Legal, Transportation). The Supplier Type Code is a unique value exported to identify supplier types.

Note:To apply one of the defined types to a supplier site, a Corporate Admin user must enable the Enable Supplier Categorization company preference.

·Units of Measure: codes used to enforce units of measure on invoice and price book line items. Entries here will appear in a drop-down menu on the supplier side. For Price Book referencing and compliance, suppliers can reference preferred and equivalent units of measure. Refer to Configuring Equivalent Units of Measure.

Note:To be applied, a Corporate Admin user must enable the Enable Units of Measure Compliance company preference.

·WO List Type: configuration of the work order (WO) line item to be a free-text field or a lookup list.

Permission:Corporate Admin

Steps:

1.Select Corporate > Workflow Configuration.

2.Click Corporate Column Configuration.

3.Click the name of the list page (link in the first column) to be configured.

4.Do one or more of the following:

§Add or remove column titles from the Configured Columns box to determine the columns that will appear on the list page.

Note:To select multiple items at once, hold the Ctrl key and select the columns. Alternatively, hold the Shift key to select a range of columns.

§Click the up and down arrows to the right of the Configured Columns box to determine the display order of your columns.

Note:Top-to-bottom order in the Configured Columns box corresponds with left-to-right order on the list page.

Configuring Taxes

This section describes the tax types and provides information for configuring your tax type.

Tax Type Profiles

·Excluded from Approval Requirements: the tax amount is excluded from the total invoice amount which determines whether a user has the appropriate authorization level to approve. If the tax has been mapped to a general ledger code, the user does not need to have the tax’s code in their approval levels to approve the invoice.

·Eligible For Early Pay Discounts: Early Pay terms are created as incentives for buyers to pay an invoice within a set number of days by reducing the amount owed. This setting applies the discount to the total invoice amount including tax.

Configuring Early Pay Programs

Early Pay programs are discounts negotiated between your company and the supplier outside of the OpenInvoice Platform. Your company may be offered a reduction on the total invoice amount if your company pays within a specified number of days after the invoice is submitted by the supplier. Early Pay programs include a dynamic discount option, which reduces the available discount using a sliding scale over a period of time. Early Pay programs are entered into OpenInvoice by buyers. Suppliers apply Early Pay programs at the time of invoice creation. The invoice displays both the original invoice amount and the invoice amount after the discount is applied.

Note:Early Pay programs are replacing the existing Early Pay terms configured in price books. Existing Early Pay price books with a supplier will continue to be used until you assign an Early Pay program to that supplier. To assign Early Pay programs, refer to Enabling and Configuring Suppliers.

Once an Early Pay program is assigned to a supplier, all price books with that supplier that include Early Pay terms are disabled. Refer to Removing Early Pay Terms on Price Books.

To configure whether the Early Pay discounts are applied to taxes, modify the Eligible for Early Pay Discounts settings on the tax type profile. Refer to Configuring Taxes.

How to Add an Early Pay Program

Permission:Early Pay Modify

Steps:

1.Select Corporate > Early Pay Programs.

2.Click Add Program.

 

3.Enter the Apply Discount of percentage.

4.Enter the Expires date (number of days after invoice submission). The maximum number of days allowed is 399.

5.Select a Status.

Note:The other fields are optional.

6.Click Save.

Early Pay Program Details

·Apply Discount of x%: discount percent to be applied to the invoice.

·Expires x days after invoice submission: number of days, from invoice submission date, in which the invoice must be paid for the discount to be applied.

·Prorate over the last x days: number of days, before the discount expires, over which the available discount will decline using a sliding scale.

·Invoices must be approved x days before discount expires: number of days, before the discount expires, in which the invoice must be approved. This represents the number of days needed to complete the payment cycles for an approved invoice.

·Discount code: an internal code used to name this program.

·Status: select Active if you want to assign this program to one or more supplier sites.

Platform Security

Users with the Corporate Admin permission are the only users with the ability to configure OpenInvoice according to your company’s business practices and workflow. This includes security configurations for user access. For instructions on how to configure corporate security preferences, refer to Security Preferences.

Considerations for Password Security

To ensure maximum security of your information on the OpenInvoice Platform, users with the Corporate Admin permission can configure password security settings for your company’s users. This includes:

·The maximum number of days before passwords expire.

·The minimum number of hours a user must wait before changing their password again.

·The number of previous passwords disallowed for re-use.

·The number of successive failed sign in attempts allowed before the user is locked out.

·The number of minutes that a locked out user must wait before attempting to sign in again.

Note:A user with the Membership Modify permission can unlock the user’s account so they can attempt to sign in again.

Considerations for the Keep Me Signed In Feature

Keep Me Signed In is the secure way to allow users to repeatedly access OpenInvoice without having to reenter their username and password. Once your company administrators determine that the feature complies with your companies internal standards and security policies, a user with the Corporate Admin permission enables and configures the feature.

The first time a user signs into OpenInvoice after this feature is enabled for your company, OpenInvoice saves a cookie on the user’s device, which makes the Keep Me Signed In checkbox available to the user on the Sign In page.

This feature does not store the username or password on the user’s device. Instead, if the user selects the Keep Me Signed In checkbox and successfully signs into OpenInvoice, an encrypted token will be generated and saved on the user’s device. Going forward, OpenInvoice will automatically use the saved token to authenticate and grant the user access to the application.

Note:This feature is device and browser specific. To use this feature across various devices and browsers, users must select the Keep Me Signed in checkbox on each device and in each browser that they use.

Note:If a user clicked Sign Out to exit OpenInvoice the last time they were working on the platform the Keep Me Signed In feature will not work the next time the user logs into the system.

Configuration Settings

Security preferences for Keep Me Signed In will allow you to configure:

·The maximum time between uses (ranging from 1 hour to 30 days) before a user’s token expires. By default, OpenInvoice sets this preference to 2 days.

·Whether this feature can be used on any device or if the device must have an allowed IP address.

·The allowable IP addresses from which users can use this feature.

Exceptions

Keep Me Signed In will stop working for a user if:

·the token expires

·the user’s password expires

·the user’s account is suspended, deleted, or deactivated

·the user’s account is locked due to multiple failed sign in attempts using another device or browser

·your company has decided to disable this feature

·the user clicked Sign out to exit OpenInvoice the last time they were working on the platform

·OpenInvoice suspects the token has been stolen
